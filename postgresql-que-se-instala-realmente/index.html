Qué se instala realmente cuando instalas PostgreSQL

datablog bases-de-datos
Actualizado: 2026-07-06

Ya tienes PostgreSQL corriendo. ¿Notaste que entraste sin que te pidiera usuario ni contraseña? Eso no es normal — y explica todo lo que sigue.

1. Instalar el software no es crear el servidor

Cuando ejecutaste brew install postgresql@17 pasaron dos cosas distintas que parecieron una sola:

  1. Se instaló el software: los programas (psql, el proceso servidor, las herramientas). Esto es solo copiar archivos — como instalar cualquier aplicación.
  2. Se fundó un servidor: se creó el cluster — el data directory con sus archivos internos, un superusuario inicial y una política de acceso. Esto lo hizo un comando llamado initdb, y Homebrew lo ejecutó en silencio, sin preguntarte nada.

La distinción importa porque el software se puede reinstalar mil veces, pero cada cluster se funda una sola vez — y en esa fundación se toman decisiones que lo acompañan toda su vida.

Una misma máquina puede albergar varios clusters (cada uno con su propio data directory, sus propias decisiones y su propio puerto si corren a la vez — así conviven dos versiones de PostgreSQL durante una migración, por ejemplo), aunque en un entorno local lo normal es tener uno solo.

2. initdb: el momento fundacional y sus tres decisiones

initdb es el comando que crea un cluster desde cero. Sus flags son, literalmente, las decisiones fundacionales:

Decisión Flag Pregunta que responde
¿Dónde viven los datos? -D (--pgdata) ubicación del data directory
¿Cómo se llama el superusuario inicial? -U (--username) el primer rol, dueño de todo
¿Cómo se autentica la gente? -A (--auth) método por defecto en pg_hba.conf

Un administrador fundando un servidor a mano en la vida real ejecutaría algo así:

initdb -D /ruta/a/los/datos -U postgres -A scram-sha-256 -W

Traducción: "los datos van aquí, el superusuario se llama postgres, todo el mundo se autentica con contraseña cifrada, y pídeme ahora la contraseña del superusuario" (-W). Existen incluso versiones más finas: --auth-local (conexiones por socket, misma máquina) y --auth-host (conexiones por red) permiten un método distinto para cada entrada.

3. Cómo tomó esas decisiones Homebrew

Homebrew ejecutó initdb por ti con estas respuestas:

Decisión Lo que eligió Homebrew Consecuencia
Data directory /opt/homebrew/var/postgresql@17 los datos son de tu usuario de macOS
Superusuario tu usuario de macOS no existe el rol postgres
Autenticación trust cualquier proceso de tu Mac entra como cualquier rol, sin contraseña

3.1 Ubicación y propiedad: dos preguntas distintas

Fíjate en un detalle de la ruta del data directory: empieza con /, no con ~. No es casualidad — esconde una distinción que te va a acompañar toda la vida con las bases de datos.

~ es un atajo de la terminal que significa "la carpeta home de quien está logueado" (/Users/tu-usuario). Solo sirve para rutas que viven dentro de tu home. El data directory no está ahí — cuelga de /opt, directamente en la raíz del disco, al mismo nivel que /Users:

/                        ← raíz del disco
├── Users/
│   └── tu-usuario/      ← esto es "~"
└── opt/
    └── homebrew/
        └── var/postgresql@17   ← el data directory

Y aquí la distinción: dónde vive un archivo y de quién es son dos cosas independientes.

  1. Ubicación: el data directory está fuera de tu home, en /opt — la zona tradicional de Unix para software instalado a nivel de máquina, disponible para todas las cuentas del ordenador.
  2. Propiedad: aun así, la carpeta pertenece a tu usuario — porque initdb corrió como tú, y quien funda el cluster es dueño de sus datos.

Por eso cualquier cuenta de tu Mac puede ejecutar psql (los binarios están en zona común), pero solo tu usuario puede arrancar el servidor y tocar los datos (la carpeta de datos es privada, con permisos que PostgreSQL exige por diseño).

3.2 Verificar las decisiones en tu instalación

Por eso entraste sin contraseña. Toda conexión a PostgreSQL empieza igual: el cliente dice "quiero entrar como el rol X". Con trust, el servidor acepta esa afirmación sin pedir ninguna prueba — ni contraseña, ni verificación de quién eres.

¿Y quién es "cualquiera"? Todo lo que corre dentro de tu Mac: una app que instalaste, un script de Python, DBeaver, un proceso de otra cuenta de usuario del mismo ordenador. Cualquiera de ellos podría conectarse declarando ser el superusuario, y el servidor lo dejaría pasar. Desde fuera de tu Mac, en cambio, nadie llega — el servidor ni siquiera escucha hacia la red externa, como verás en la pregunta 4 (¿en qué direcciones y puerto escucha el servidor?). Es cómodo para aprender, pero es una puerta sin cerradura por dentro.

Recién instalado el servidor, hay cuatro preguntas que puedes (y deberías) hacerle. Todas se responden desde psql:

Pregunta 1 — ¿Dónde viven los datos y la configuración?

psql -d postgres -c "SHOW data_directory;" -c "SHOW hba_file;"

El data directory que ya conoces, y dentro de él pg_hba.conf — el archivo con las reglas de acceso.

Pregunta 2 — ¿Qué roles existen?

psql -d postgres -c "SELECT rolname, rolsuper FROM pg_roles WHERE rolname NOT LIKE 'pg_%';"

La t y la f de la salida son true/false — así muestra psql los booleanos. Recién instalado verás un solo rol: el tuyo, con rolsuper = t (quien funda el cluster es superusuario), y ningún rol postgres a la vista. El filtro NOT LIKE 'pg_%' esconde los roles internos de mantenimiento que PostgreSQL trae de fábrica.

Pregunta 3 — ¿Qué reglas de acceso están activas?

grep -v "^#" $(psql -d postgres -tAc "SHOW hba_file;") | grep -v "^$"

El comando lee pg_hba.conf quitando comentarios y líneas vacías — quedan solo las reglas vivas. Cada línea es una puerta del servidor, descrita en columnas: entrada → base → rol → [dirección]→ método. Hay dos tipos de entrada: local (el socket — conexiones desde tu propia terminal, misma máquina) y host (conexiones por red).

En las puertas de red aparecen dos direcciones raras: 127.0.0.1/32 y ::1/128. La idea detrás es simpática: toda máquina tiene una dirección reservada para hablarse a sí misma, llamada localhost.

Cuando un programa la usa, el mensaje "sale por la red" y vuelve a entrar al instante, sin tocar ningún cable ni el WiFi — es la entrada local, pero disfrazada de conexión de red. ¿Y por qué dos direcciones para la misma idea? Porque en tu máquina conviven los dos idiomas de direcciones de internet, y localhost se escribe distinto en cada uno: 127.0.0.1 en el clásico (IPv4) y ::1 en el moderno (IPv6) — una puerta por idioma, para que nadie se quede afuera hable el que hable. El número tras la barra es la precisión de la regla: /32 y /128 significan "exactamente esta dirección, ni una más".

¿Y las líneas replication? Son esas mismas puertas, pero para un visitante muy específico: otro servidor PostgreSQL que viene a clonarse de este en tiempo real (así se montan las réplicas de respaldo en producción). En tu máquina de aprendizaje, nadie toca esas puertas.

La última columna es el candado de cada puerta, y hay tres métodos que debes conocer:

  1. trust — sin candado: el servidor acepta cualquier identidad sin prueba. Es lo que verás en toda la salida recién instalado, y no debería sobrevivir a este post.
  2. peer — el candado correcto para la puerta local: el sistema operativo da fe de quién eres (solo entras como el rol que coincide con tu usuario), sin contraseñas.
  3. scram-sha-256 — el candado correcto para las puertas host: contraseña cifrada, porque en una conexión de red el sistema operativo ya no puede dar fe de quién está al otro lado.

Esa es exactamente la configuración objetivo que montarás en la sección 6: peer en local, scram-sha-256 en las host, cero trust.

Pregunta 4 — ¿En qué direcciones y puerto escucha el servidor?

psql -d postgres -c "SHOW listen_addresses;" -c "SHOW port;"

La salida trae dos datos que parecen lo mismo pero responden preguntas distintas:

  1. listen_addresses = localhost responde ¿desde dónde se puede llegar al servidor? Piensa en la dirección como el edificio: localhost significa que el servidor solo atiende visitas originadas en tu propia máquina. Para cualquier otro ordenador de tu red WiFi, tu PostgreSQL no existe — no es que rechace conexiones, es que no hay puerta hacia afuera.
  2. port = 5432 responde ¿en qué puerta del edificio está PostgreSQL? Tu máquina corre muchos programas que escuchan conexiones a la vez, y cada uno toma un número de puerta distinto — el puerto — para que los mensajes lleguen al programa correcto. El 5432 es el número tradicional de PostgreSQL (igual que el 443 es el de los sitios web seguros).

Juntos se leen: "a PostgreSQL se llega solo desde esta máquina (dirección), tocando la puerta 5432 (puerto)". Por eso los clientes como DBeaver piden host y puerto por separado — son las dos coordenadas de la conexión. A diferencia de todo lo anterior, esta configuración no vive en pg_hba.conf sino en postgresql.conf, el otro archivo del data directory — y en la sección 5 verás por qué son dos capas de seguridad distintas.

4. Cómo pasa en la vida real (fuera de tu Mac)

initdb corre siempre y exactamente una vez por cluster — lo que cambia entre entornos es quién lo ejecuta y qué decisiones toma:

Entorno ¿Quién ejecuta initdb? Superusuario Autenticación por defecto
Homebrew (Mac) automático, al instalar tu usuario de macOS trust (abierto)
Ubuntu/Debian (servidores reales) automático, al instalar postgres peer local + scram-sha-256 por red
Red Hat/CentOS tú, manualmente (postgresql-setup --initdb) postgres peer local + scram-sha-256 por red
Docker automático, en el primer arranque postgres te obliga a definir la contraseña (POSTGRES_PASSWORD); método scram-sha-256
Nube gestionada (Cloud SQL, RDS, Supabase) invisible, lo hace el proveedor postgres (o similar) la contraseña la defines en la consola; método scram-sha-256

Dos observaciones de esta tabla:

  1. El origen del misterio -U postgres (nombre usuario -> postgres): casi todos los tutoriales dicen psql -U postgres porque en Linux, Docker y la nube el superusuario se llama así — en Linux, initdb lo ejecuta un usuario de sistema llamado postgres creado para eso. Homebrew es la excepción: lo ejecutaste tú, así que el superusuario eres tú (tu nombre de usuario).
  2. Homebrew es el único que deja la puerta abierta: todos los entornos serios nacen con autenticación real. Docker directamente se niega a arrancar sin contraseña. Por eso en una sección posterior de este post cerraremos esa puerta.

5. Las dos capas de acceso: postgresql.conf y pg_hba.conf

Las preguntas 3 y 4 de la sección 3.2 (¿qué reglas de acceso están activas? y ¿en qué direcciones y puerto escucha el servidor?) no eran dos datos sueltos — son las dos capas con las que PostgreSQL decide quién entra, cada una con su propio archivo de configuración (ambos viven en el data directory):

Capa Archivo Pregunta que responde
1. ¿Qué puertas existen? postgresql.conf → listen_addresses en qué direcciones escucha el servidor
2. ¿Quién pasa por cada puerta, y con qué prueba de identidad? pg_hba.conf el reglamento del portero, línea por línea

Puedes interrogar las dos capas en cualquier momento (con los mismos comandos que ya usaste para verificar las decisiones de Homebrew en la sección 3.2):

# Capa 1 — ¿en qué fachadas hay puerta?
psql -d postgres -c "SHOW listen_addresses;"

# Capa 2 — ¿qué dice el reglamento del portero?
grep -v "^#" $(psql -d postgres -tAc "SHOW hba_file;") | grep -v "^$"

Y la forma más fácil de no confundirlas nunca: listen_addresses habla de tus fachadas; pg_hba.conf habla de tus visitantes.

5.1 Capa 1 — listen_addresses: tus fachadas

Tu máquina tiene más de una "fachada" por donde recibir conexiones: la interna (localhost, la dirección con la que se habla a sí misma) y la que da a tu red (la IP que el router le asignó, algo como 192.168.1.50). listen_addresses decide en cuáles de tus propias fachadas el servidor construye una puerta:

listen_addresses = 'localhost'                  # hoy: solo fachada interna
listen_addresses = 'localhost, 192.168.1.50'    # + puerta hacia tu red WiFi
listen_addresses = '*'                          # puertas en todas las fachadas

Fíjate en lo que no aparece ahí: ninguna dirección de visitantes. Esta capa no decide quién entra — solo dónde hay puertas. Con localhost, para cualquier otra máquina de tu red tu PostgreSQL no existe: no es que rechace conexiones, es que no hay puerta que tocar.

Detalle operativo: cambiar listen_addresses requiere reiniciar el servidor (brew services restart postgresql@17) — construir puertas nuevas no se hace con el edificio en marcha.

5.2 Capa 2 — pg_hba.conf: tus visitantes

De las puertas que existen, este archivo decide quién cruza cada una y con qué prueba de identidad — es el reglamento del portero que ya leíste en la pregunta 3 (¿qué reglas de acceso están activas?). Cada regla del archivo es siempre la misma frase con cinco espacios en blanco:

¿por qué entrada? + ¿a qué base? + ¿con qué rol? + ¿desde qué dirección?  →  ¿qué prueba exijo?

Y ojo con un malentendido común: el reglamento no es una lista de invitados. Una lista de invitados solo respondería una de las preguntas ("¿con qué rol?"); aquí el rol es apenas una de las cuatro condiciones de cada regla — de hecho, en tu servidor recién instalado todas las reglas dicen all en esa condición: no filtran por quién eres, filtran por dónde entras y desde dónde vienes. En producción sí verás reglas que nombran roles concretos (ahí el portero se pone en modo lista de invitados), pero siempre como parte de la frase completa.

Aquí sí aparecen direcciones de visitantes: si un día quisieras que otra máquina de tu red entre, además de abrirle la fachada en la capa 1, necesitarías una línea que la reciba:

host    youtube_analytics    datablog    192.168.1.0/24    scram-sha-256

Se lee: "por la puerta de red, a esta base, con este rol, pueden entrar visitantes de mi red WiFi (las 256 direcciones del rango /24), probando identidad con contraseña".

Dos reglas para leer este archivo como un administrador:

  1. Si una conexión no coincide con ninguna línea, se rechaza. No hay puerta por defecto — todo lo que entra, entra porque una línea lo permite.
  2. Gana la primera línea que coincide — se lee de arriba hacia abajo, el orden importa.

Detalle operativo (y contraste con la capa 1): los cambios en pg_hba.conf no requieren reinicio — basta recargar con SELECT pg_reload_conf();.

En un servidor de producción las dos capas se estiran al máximo: listen_addresses = '*' (las apps se conectan desde otras máquinas — las fachadas deben existir) y entonces pg_hba.conf se convierte en la muralla crítica: rangos de IP concretos, bases concretas, siempre contraseña cifrada. En tu Mac es al revés: la capa 1 ya te aísla del mundo, y la capa 2 es la que hoy está en trust y toca cerrar.

6. Cerrar la puerta: el cambio, paso a paso

El plan para la capa 2: peer para lo local (el sistema operativo da fe de quién eres — el estándar de los servidores Linux) y scram-sha-256 para la red (contraseña cifrada — el estándar de Docker y la nube).

# Paso 1 — localizar el archivo
psql -d postgres -tAc "SHOW hba_file;"

# Paso 2 — respaldarlo antes de tocar nada
cp <ruta-del-archivo> <ruta-del-archivo>.bak

Paso 3 — crear las contraseñas. Va antes de cerrar nada por una razón: la llave debe existir antes de instalar la cerradura — si activas scram sin contraseñas creadas, ningún cliente TCP (como DBeaver) podrá entrar:

psql -d postgres
# dentro de psql:
\password              # contraseña para tu rol (la pide dos veces)
\q

# Paso 4 — editar el archivo (nano, VS Code, el editor que uses)

#Si tienes el comando code activo 
code /opt/homebrew/var/postgresql@17/pg_hba.conf

# Cambiar el método de las líneas que empiezan con "local":  trust → peer
# Cambiar el método de las líneas que empiezan con "host":   trust → scram-sha-256

Paso 5 — recargar y probar. No hace falta reiniciar el servidor:

psql -d postgres -c "SELECT pg_reload_conf();"

# Probar 1: tu conexión local sigue entrando sin contraseña (peer: macOS da fe de ti)
psql -d postgres -c "SELECT current_user;"

# Probar 2: intentar entrar como un rol ajeno debe FALLAR — esta es la señal de éxito
psql -U otro_rol -d postgres
# → FATAL: Peer authentication failed  ✓ el portero ya verifica identidades

# Probar 3: por red (como DBeaver) ahora pide contraseña
psql -h 127.0.0.1 -d postgres

Con esto tu instalación local pasa de "modo juguete" a la configuración que encontrarías en un servidor real: la misma que los DBA montan a mano cuando fundan un cluster.

7. Lo que acabas de lograr

Tu instalación pasó de "modo juguete" a la configuración que encontrarías en un servidor real — la misma que los DBA montan a mano cuando fundan un cluster. Pero lo importante no es solo el cambio; es todo lo que ahora entiendes de tu servidor:

  1. Instalar no es fundar: sabes que detrás de brew install corrió initdb, y que sus tres decisiones (data directory, superusuario, autenticación) explican los tres "misterios" de tu instalación — por qué entrabas sin contraseña, por qué no existe el rol postgres, por qué los datos viven bajo tu usuario.
  2. Sabes interrogar cualquier servidor PostgreSQL: las cuatro preguntas de la sección 3.2 funcionan igual en tu Mac, en un Linux de empresa o en Docker.
  3. Distingues las dos capas de acceso: listen_addresses (tus fachadas) y pg_hba.conf (tus visitantes), cada una con su comando de verificación y su forma de recargarse.
  4. Cerraste la puerta con los métodos del mundo real: peer donde el sistema operativo puede dar fe de ti, scram-sha-256 donde no — y comprobaste con tres pruebas que la seguridad quedó activa sin estorbar tu trabajo diario.

8. Qué sigue

El portero ya verifica quién entra. La siguiente pregunta natural es la otra mitad del control de acceso: una vez adentro, ¿qué puede hacer cada quien?

Hoy tu cluster tiene un solo rol: tú, superusuario absoluto. Pero en el mundo real nadie trabaja así — las aplicaciones y los analistas usan roles con permisos mínimos (leer solo lo que necesitan, sin poder crear ni borrar nada). Crear un usuario de solo lectura con GRANT, y entender por qué un superusuario ignora todos los permisos, es el tema del siguiente post de la serie.

9. Gotchas

  • initdb te avisó y nadie lo lee: cuando se usa trust, initdb imprime una advertencia explícita de que es inseguro. Homebrew la muestra durante la instalación — enterrada entre decenas de líneas de output.
  • Desinstalar PostgreSQL con brew no borra el cluster: brew uninstall elimina los binarios pero el data directory queda intacto en /opt/homebrew/var/. Reinstalar reutiliza el cluster viejo con todas sus decisiones fundacionales — si quieres empezar de cero de verdad, hay que borrar también esa carpeta.
  • peer amarra rol ↔ usuario del sistema operativo: tras el cambio, desde tu sesión solo puedes entrar por socket como el rol que lleva tu nombre. Para conectarte como otro rol desde la misma cuenta, usa la vía de red con contraseña: psql -h 127.0.0.1 -U <rol>.
