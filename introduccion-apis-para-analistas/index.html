Introducción a APIs para analistas de datos

1. El concepto

Una API es un contrato entre dos sistemas: define qué se puede pedir, cómo pedirlo y qué se recibirá a cambio. No es un archivo ni un servidor — es un conjunto de reglas.

Como analistas, nuestro rol es consumir APIs para extraer datos. No las construimos ni las integramos entre sistemas — las usamos para pedir datos, almacenarlos y analizarlos. Ese es el punto de partida.

¿A qué datos puedo acceder?

Antes de hacer cualquier petición hay que saber si la API existe y cómo conseguir acceso. Depende de a quién pertenecen los datos:

  1. Datos de empresa — SAP, Oracle, Azure, GCP, AWS. Los controla la organización. Necesitas que alguien interno (IT o desarrollo) te dé acceso.
  2. Datos propios — Google Workspace, Notion, GitHub. Son tuyos. Puedes generar las credenciales directamente desde el panel de la plataforma.
  3. Datos de prueba — entornos sandbox que la plataforma ofrece para desarrollo. Stripe, Salesforce Developer Edition, HubSpot. No son datos reales — sirven para aprender y probar.

Cómo hacer una petición

Para hacer una petición necesitas tres cosas:

  1. Una herramienta — no siempre es código. El navegador funciona con APIs públicas o las que exponen el API key en la URL. Postman es ideal para explorar sin código. Python con requests es para cuando ya sabes qué pedir y quieres automatizarlo.
  2. La documentación — el manual que te dice qué endpoints existen, qué parámetros acepta cada uno y qué te devuelve. Sin ella estás adivinando.
  3. El método correcto — GET para pedir datos (el más común), POST para crear o enviar parámetros complejos, PUT/PATCH para actualizar, DELETE para eliminar.

Autenticación

Antes de recibir datos, la API necesita saber quién eres. Los mecanismos principales:

  • API Key — credencial estática que alguien genera y te pasa directamente. Permanente hasta revocación manual. Algunas plataformas permiten configurar permisos, otras no.
  • OAuth 2.0 — protocolo estándar que siempre requiere una aplicación registrada en la plataforma. Genera tokens temporales que expiran. Tiene dos flujos: con usuario (el usuario autoriza en pantalla) y sin usuario — Client Credentials (máquina a máquina, sin login).
  • Otros — Basic Auth, Bearer Token, JWT, HMAC. Menos frecuentes pero presentes en sistemas legacy o empresariales.

La diferencia clave entre API Key y OAuth no está en los permisos — ambos pueden tenerlos. Está en la permanencia (API Key es estática, OAuth genera tokens que expiran) y en la identidad (API Key es una credencial suelta, OAuth siempre tiene una aplicación registrada detrás).

Qué devuelve una API

Toda respuesta tiene dos partes:

  1. Status code — el número que dice si funcionó o no. 200 es éxito, 401 es sin autenticación, 403 es sin permiso, 404 es endpoint inexistente, 429 es rate limit superado, 500 es error del servidor.
  2. Cuerpo — los datos en sí, casi siempre en JSON. Algunas APIs devuelven XML o CSV.

3. Mi conclusión

Las APIs no nacieron para analistas — nacieron para conectar sistemas de software. Pero casi todas las permiten consumir para extraer datos. Entender eso cambia cómo te posicionas: no eres el caso principal, eres un caso tolerado. Por eso el rate limiting duele más al analista que a cualquier otro usuario.

Lo que sí es tuyo es el flujo: identificar si la API existe, conseguir acceso, leer la documentación, autenticarte y hacer la petición. Ese flujo es el mismo independientemente de la plataforma.
